coreutil 阅读笔记(一)

发表于 2015-09-22   |  

coreutil是什么

coreutil全称是GNU core utilities, 是linux下一些基础工具/命令的实现，包括像耳熟能详的ls, cat, rm等等。这些工具有一些是对部分系统调用的封装，还有一些是简单的文本处理等等。APUE在介绍系统调用时，有时就会编写一些类似的小工具做为示例，比如介绍文件目录相关内容时，会实现一个简单的ls。但是实际使用的ls，会非常复杂，支持众多的命令行选项，事实上，coreutil里ls实现包含了将近5000行的代码。

决定读这部分代码主要是出于几个原因

  • 熟悉常见系统调用的使用，不管是APUE 还是TLPI ,毕竟还是手册类型的，读起来比较枯燥。
  • 借鉴学习，不管是ls,rm,还是pwd，date等等，每天都会被各种使用，同时支持众多选项，这些命令的实现应该是比较精巧的。
  • 顺便回顾下这些命令的常见使用方式/选项，写shell的时候可能也会更得心应手些。

准备工作

通过以下方式获得coreutil最新的源码

coreutils/src目录下，包含了120多个.c文件，基本上每一个.c对应于一个命令，比如ls.c对应ls的实现，pwd.c对应pwd的实现。共同依赖一些基础的头文件。各个.c之前基本上是彼此独立的，没有依赖。读起来很方便。

编译这些文件也非常简单, configure之后make即可，src目录下会生成一堆可执行文件。直接运行即可。

由于文件之间彼此独立，没有太多的依赖，那就按照字母序来吧。

先看这几个

  • base64.c
  • basename.c

base64.c

作用

这个编译后生成一个base64的二进制文件，作用是进行base64编码/解码工作

base64是一种编码标准，主要目的就是把一段内容（包括文本，包括二进制），变成一种”文本”格式，编码之后，仅会包含A-Z a-z 0-9 + /以及 = 这65种字符构成的序列。A-Z a-z 0-9 + / 这些字符分别对应于0-63这64个位置。=只会用于最后的填充

具体的编码方式很简单，将输入的字符流，每3个Byte一组，最后一组如果不足3个Byte，用0补齐（不是数字0，是’\0’)。3个Byte对应24个bit，将这24个bit从高位到低位重新分成4个部分（每个部分包括6bit），于是每个部分（6个bit）的十进制落在了[0-63]这个范围内，正好对应前面的码表，从码表中取出相应的字符即可。最后填充的’\0’用=编码

比如原始内容是两个字符，’ab’（不包含两边引号），编码过程如下

  1. 填充1个0，即’a’,’b’,’\0’
  2. 原始内容 97 98 0 变成24bit 01100001 01100010 00000000
  3. 重新切成4个部分, 011000 010110 001000 000000
  4. 得到相应的位置，24 22 8 0
  5. 查码表，得到编码后的结果 YWIA
  6. 最后的’\0’用=代替，即最终结果为 YWI=

类似的，单个a编码后，结果是YQ==

实现

  • 一些通用的初始化工作

    1. initialize_main(&argc, &argv), 这个宏定义在src/system.h里，一般没什么用，可能在特定的VMS里边才会用到
    2. set_program_name(argv[0]), libtools生成的临时的可执行文件名，有时会带上/.lib或者/.libs/lt-的前缀，去掉这些前缀。
    3. setlocale, bindtextdomain, textdomain 一些本地化的设置。linux使用locale来进行本地化的设置。面向用户的输出，用_函数包裹起来(实际是gettext)，src/po目录下，定义了各种输出对应的各种语言的表示。
    4. atexit(close_stdout), exit时关闭stdout

  • 命令行参数处理,

    1. -d decode=true
    2. -w 设置打印时一行多少字符
    3. -i decode时是否要忽略非base64表里的字符
    4. 得到文件名，如果没有文件名，或者文件名是-, 则从stdin读取
    5. fadvise, 提示kernel，准备读文件了，顺序读。
    6. 开始decode或者encode, 进入真正的处理逻辑
      1. do_decode(input_fh, stdout, ignore_garbage)
      2. do_encode(input_fh, stdout, wrap_column)

  • do_encode

    1. 每次尝试读ENC_BLOCKSIZE(30KB)的数据, 显然每次读的长度应尽可能是3个倍数，同时不能太少。
    2. 如果读到了至少1个Byte
      1. base64_encode(inbuf, inlen, outbuf, outlen), $outlenn = (inlen+2)/3 *4$
      2. wrap_write, 格式化输出

    3. 如果发现inlen = outlen/4*3, 即不需要补0的情形, 这个处理就简单了(b64c就是那个码表)

    4. 否则处理的时候，考虑了outbuf里空间不够的情形。空间不够了，立刻退出，避免缓冲溢出（虽然outlen其实是能够提前算好的）。

      1. 还是3个输入字符一组
      2. 每组输出的前两个字符不用考虑=情形
      3. 从三个字符（对应输入的第二个字符），需要考虑补0的情形
      4. 全部结束后，如果outbuf里还有空间，补上最后的字符串结尾的’\0’
    5. 从outbuf里输出
      1. wrap_column = 0, 简单情形，直接fwrite(outbuf, 1, len, stdout) 即可
      2. 否则，每行输出wrap_column个字符，直至完成
  • do_decode
    1. 判断是否需要忽略非base64的字符(ignore_garbage), 如果是，看到一个非法字符，则跳过这个字符，将剩下的部分memmove过来。这个实现是有点怪异的。因为如果有N个非法字符，岂不是要memmove N次？直接一遍扫描完标记住，一次性move不会更快些吗？这个我在Stackoverflow上提了一个问题 期望有人能够解释下原因。
    2. 调用base64_decode_ctx完成实际的工作
      1. 调用decode_4, 每次decode4个字符, decode的逻辑和encode正好相反, 注意结尾=的处理
    3. 其他没有太多特殊的
  • 一些收尾的工作

basename.c

这个对应于basename命令，作用就是对于一个包含路径的文件名，移除路径，也按照根据选项可以移除指定的后缀名。实现也比较简单，可以认为就是最基础的字符串的处理。

实现

  • 一些通用的初始化工作(initialize_main, set_program_name, setlocale, bindtextdomain, textdomain, atexit)

  • 命令行选项的处理

    1. -s及定义的后缀 决定是否去掉指定的后缀
    2. -a 是否处理多个输入
    3. -z 每次输出时 是否需要带换行

    4. 命令行选项处理完后，获取剩下的输入 若多余1个，判断有-a. 这里的实现似乎有bug，按理说若没有-a，输入两个路径也应该报错，但其实会正常执行第一个，忽略第二个，只有输入>=3个时，才会报extra operand错误,似乎应该是 optind + 1 &lt argc

  • 逐一处理每个输入 perform_basename(argv[optind], suffix, use_nuls)

    1. name = base_name(string)
      1. base = last_component (name), 挨个检查，找到最后一个slash, 将剩下的部分返回
      2. 去掉末尾的slash
      3. 如果需要去掉后缀，再去掉后缀

cat.c

这个对应的就是天天使用的cat命令。连接文件并在终端上显示。但是具体实现的时候有一些细节还是值得学习的。

实现

  • 通用的初始化工作
  • 命令行选项的处理
    1. 是否显示行号
    2. 是否显示空行的行号
    3. 是否将多个空行压缩为一行
    4. 是否显示行末
    5. 是否显示不可打印的字符
    6. 是否显示tab
  • 对于不带任何选项的情形，使用simple_cat直接原样输出
  • 否则将各种需求传到cat函数，格式化后输出

实现时，有几点需要注意一下

  • linux各种设备，管道等都是视为文件的，因此所谓输出其实就是fread（从磁盘文件或者标准输入）一块内容至内存(inbuf)，然后再将这块内容格式化后放到一块内存上(outbuf)，最后批量fwrite到文件（终端就是STDOUT）的过程
  • 读文件一般都是一次性读一大块的内容。cat在实现时，采用的是一个经验值128KB,这也是实验出的结果
  • 写到内存时，从内存分页的起始写，效率会更高一些。getpagesize()获得一个分页的大小，下面的ptr_align可以从ptr位置开始往后找到第一个分页处（alignment表示每一个分页大小，在我的系统上是4KB，即4096个字节）。因此在申请读入内存时，考虑到了偏移，使用的是 128KB + 4096

chcon.c

这是一个和SELinux相关的权限控制的一个命令。SELinux可以对应用程序的权限做更精细的控制。开启SELinux后，这个命令用于改变文件security context。但由于一般机器都不会开启SELinux功能，因此这个命令使用的几率也就很小。这里就直接忽略了。

chgrp.c

对应chgrp命令，作用就是改变文件所属的组。

具体实现过程如下

  • 初始化，解析命令行选项

  • 获得文件当前的group_id，并转成group_name

  • 实际操作在chown-core.c里的chown_files完成, 遍历时，使用的是FTS系列的API，没有采用POSIX标准下的nftw那一套。

  • POSIX里明明定义了chown/lchown/fchown系列函数用于改变文件的owner和group，但是coreutil里边似乎没有采用这套API, 而是调用了lchownat

chmod.c

对应chmod命令，用于改变文件权限(R/W/X)

实现如下

  • 初始化，解析命令行选项
  • 核心处理在process_file(FTS *fts, FTSENT *ent)里边
  • 遍历时，处理一些特殊情况，正常文件，调用chmodat完成实际的操作

chown.c

对应chown命令，改变文件所属。这个和chgrp很类似。底层调用的函数也是一样的。
