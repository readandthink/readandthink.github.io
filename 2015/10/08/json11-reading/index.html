json11 阅读笔记

发表于 2015-10-08   |  

json11是什么

json11是dropbox公司提供的解析json的C++开源库。和另外两个知名库(rapidjson, jsoncpp)相比，json11由于大量使用了C++11的很多新的特性，代码写的非常简洁(json11源码只有900多行，相比之下，rapidjson实现超过了5000行)，接口也很清晰。

快速使用

下面这个示例，展示了如何使用json11将一个源字符串解析为json object，并在此基础上做各种修改，最后重新序列化。如果只是使用，这个例子(test.cpp)应该已经是比较完整了。

编译运行

可以看到，json11的接口十分清晰，除了在修改时需要注意，使用const_cast去掉默认的const属性，其他在使用上是比较直观方便的, 另外json11还支持自定义的类型（定义好to_json方法即可），可以参考官方给的示例

实现

json11源码主要包含了一个头文件(json11.hpp)和一个cpp文件(json11.cpp）。

  • 头文件里定义了对外的接口
  • cpp文件里则是具体的实现

我们先看头文件

json11.hpp: 对外的API

initializer_list 这个头文件里定义了类模板initializer_list, 这是一种轻量级的表示初始化列表的类模板

再往下

这里有两点需要注意

  • Json类，对应于一个具体的完整json对象，注意到根据ECMA-404, 除了通常意义上的json object外(比如{“a”: 10, “b”:null}这种）普通的一个数字，或者字符串字面量等等也视为一个合法的json对象
  • final是C++11里新增的关键字，用于表示所修饰的类不能够再被继承。

继续

json value一共包含以下几种

  • string
  • number
  • object
  • array
  • true
  • false
  • null

和这里定义的Type正好匹配

然后对外定义了array和object类型。实际就是vector和map

然后是一系列构造函数，class Json可以通过ECMA-404里定义的七种类型任何一种方式初始化。

除此之外，对于用户自定义的类型，只要包含了to_json方法，也可以用来进行初始化

注意到判断某个类型是否有to_json方法，实际上用到了SFINAE的trick,如果某个类型没有to_json, 则decltype(&T::to_json)会推导失败，于是不会匹配到这个构造函数上，如果所有的构造函数都匹配不上，编译出错。

除此之外，对于map-like(比如std::map, std::unordered_map等）或者vector-like的类型（比如std::list, std::vector, std::set等），同样可以用来初始化

继续往下走

这里涉及到C++11的新增的一个语法delete function,通过显示的禁止使用void *类型，避免隐式转化为一个bool类型。

接来下，就是定义了type()函数，获得当前这个对象实际类型。以及一系列is_sometype函数，方便使用，包括：

  • is_null()
  • is_number()
  • is_bool()
  • is_string()
  • is_array()
  • is_object()

然后是获得对象实际的值的一堆函数

  • number_value()
  • int_value()
  • bool_value()
  • string_value()
  • array_items()
  • object_items()

注意到这里对于非基本类型（非POD类型），返回的都是const reference，默认是不允许修改的。若实在需要修改，需要在外面显示的去掉const属性

对于array或object类型，重载下标索引运算符是很有必要的，类似的，这里返回的const reference, 由于数组或者object里值可以是任意类型，因此这里返回的正是Json类型的常引用。

然后定义了序列化函数dump，用于将当前这个Json对象序列化到字符串里。以及反序列化函数parse，用于从字符串里获得Json对象。这里如果parse失败，错误信息会写到传入的err里边。除此之外，还定义了一个辅助的parse_multi函数，方便一次性解析多个字符串。

再往下，定义了比较运算符的重载，需要做大小或者相等判断时，会比较方便。需要注意的是，类型不同的Json对象，默认彼此都是不相等的。

最后还有一个has_shape函数，用于判断当前这个Json对象，是否符合给定的”形状”, 而”形状”则是通过 key - type 这么一个列表来表示的。

Json类型只有一个私有的成员变量，是一个指向内部实际值类型JsonValue的智能指针

JsonValue是一个内部使用的类型，表示对各种Json类型的值的抽象。提供了一些诸如获取实际类型，获取实际值等这样的抽象的接口。具体实现，当然需要由实际继承的各个值类型来实现。

json11.cpp: 内部的实现

首先定义了一个常量max_depth, 表示最多处理的嵌套的深度, 默认为200。这个绝大多数情况下，是足够了。

然后是一些序列化的函数，利用重载针对各种类型来进行处理

  • null类型，直接序列化为字符串”null”
  • double类型，只要不是inf或者nan，正常转成字符串，否则转换为”null”
  • int类型，正常转为字符串（snprintf)
  • bool类型，序列化为”true”或者”false”
  • 字符串类型。简单字符串当然可以前后加上引号之后，直接返回，但是当涉及到一些控制字符时，就需要单独处理了
  • array类型，就是对各个元素进行序列化，然后前后加上[],元素间用逗号分隔
  • object类型，就是对各个key-value序列化，前后加上大括号{}, key value间增加冒号:, kv之间逗号,分隔

然后就是对各个值类型的定义。在头文件里定义了抽象类JsonValue

在JsonValue基础上定义了Value类型，Value类是一个模板类，包含两个模板参数，一个是Json::Type描述的类型，一个是对应的实际类型。针对各种情形，各种实际类型从Value的对应的特化的类里边继承得到。

针对各种实际类型，定义了各个对应的静态全局变量，表示各自默认的取值。

紧接着，就是对Json类里各个构造函数的实现了。其实没太多好说的，注意到非基本类型（string，array，object）同时定义了move构造函数。之后就是Json类里取得实际值得各个函数的实现。实际都是通过m_ptr指向的相应的JsonValue获得的。相应的实现也都在对应的JsonValue衍生的各个子类里。

然后就是比较函数(==和<)的实现（其他比较函数都可以根据这两个得到），可能需要注意的是，类型不同的Json对象一定是不相等的

再往下，就是实际的反序列化（parse）的逻辑了，这部分代码比较核心，也更复杂些。

所谓parse，就是从一段字符串里得到相应的Value的过程。我们具体来看是怎么进行parse的

首先定义了两个辅助函数

  1. string esc(char c): 这个函数作用就是判断char c是否可打印，如果是普通可打印的字符，正常打印，否则打印字符的ascii码。这个在后续打印错误信息的时候会用到
  2. bool in_range(long x, long lower, long upper): 这个函数也很简单，就是判断$\text{x}是否满足$ \in [\text{lower}, \text{upper}]$

然后定义了JsonParse这个结构体，保存当前解析状态，以及实现具体的解析动作

  1. str是一个指向实际待解析的外部字符串的引用。
  2. 具体解析时，是逐个字符处理的，i表示下一个待处理的字符的index
  3. err保留解析时，遇到的错误
  4. 出现任何错误，failed置true

可以看到，出现任何错误了，failed置为true，相应错误信息写到err里边。

然后继续是两个函数，这两个函数都比较简单

  1. consume_whitespace: 从当前为止往下找，直到找到第一个非空字符(不是’ ‘, ‘\r’, ‘\n’, ‘\t’)返回
  2. get_next_token: 利用上边这个函数，得到下一个非空字符

下面就是unicode字符的处理。注意到unicode定义了多个“平面(plane)”, plane0也被称作Basic Multilingual Plane(BMP), 这里边的字符可以通过\u后边跟上4个[0-9a-fA-F]的字符构成，其他plane的字符，ECMA-404规定，使用两组这样的\u表示(即12个字符)

To escape a code point that is not in the Basic Multilingual Plane, the character is represented as a twelve-character sequence, encoding the UTF-16 surrogate pair. So for example, a string containing only the G clef character (U+1D11E) may be represented as “\uD834\uDD1E”.

unicode是对字符集的规定。而utf-8则是具体的编码方式。理论上，经过utf-8编码后，一个字符可能会占1~6个字节（视这个字符在unicode码表里的具体位置而定），但是根据ECMA-404的规定，BMP平面外的字符，使用专门的编码方式（拆成两个unicode，对应编码后，占用两个1-4字节的utf-8）

具体的parse过程，在 parse_json里实现。是一个典型的递归的过程。

  1. depth表示当前嵌套的层数，嵌套层数超过预设的层数(200)了，直接fail然后退出。
  2. 尝试读取下一个非空字符
  3. 如果发现当前这个字符是’-‘或者0~9,认为当前要解析的是一个数字。调用parse_number尝试进行解析
  4. 如果发现当前这个字符是’t’, 认为当前要解析的是boolean true
  5. 如果发现当前这个字符是’f’, 认为当前要解析的是boolean false
  6. 如果发现当前这个字符是’n’, 认为当前要解析的是null
  7. 如果发现当前这个字符是’”‘, 认为当前要解析的是一个字符串，调用parse_string尝试进行解析
  8. 如果发现当前这个字符是’{‘, 认为当前要解析的是一个object
  9. 如果发现当前这个字符是’[‘, 认为当前要解析的是一个array
  10. 其他情况，说明发生了错误，直接fail然后退出。

对于每种解析，我们来逐一解释

  1. parse_number，数字的解析
    json里的数字，完整的格式如下

    一个number最多会包含四个部分

    1. 符号（+，-，或者没有）
    2. 整数部分,整数部分不能包含前置0, 即034这种形式非法
    3. 小数部分,由小数点和十进制数字构成

    4. 指数部分, e/E 后边仅若干个十进制数字

      实际从字符串表示转换为number，使用的是std::atoi或者std::strtod。以下是完整的解析过程，代码写的很精炼（感觉可以作为一道区分度很高的面试题了）

  2. true/false/null 的解析非常trival，不再赘述

  3. parse_string，字符串的解析
    我们先看ECMA-404标准里，对于json字符串的格式要求和说明。下图展示了一个合法字符串的形式


    有了这个图，再看parse_string函数，就比较容易理解了。

    1. 首先如果发现当前字符是’”‘，表示字符串解析结束了（\”的情况会在后边专门处理）
    2. 如果当前字符在[0,0x1f]这个范围内，说明出现了某个不可见的控制字符，而在序列化后的json里，这样的控制字符，应该以可读的方式呈现的(比如\r,\n等等），出现则说明发生了错误，直接退出
    3. 对于其他普通并且非转义字符(), 原样打印
    4. 转移字符单独处理
      1. \u, 说明接下来是一个unicode码表里的字符，将其转为utf-8编码后输出
      2. 其他控制字符，包括\b,\f,\r,\n,\t,\”,\,\/, 相应进行处理（比如原来两个字符’\’和’n’, 变成一个字符’\n’等等）
      3. 其他字符，说明跟的是一个非法的转义，直接fail

  4. 对object的处理
    这个是一个典型的递归过程

    1. 首先判断当前这一层，是不是一个空的object，如果返回一个空的map(然后被隐式转为相应的的Json对象）
    2. 然后是一个循环，分别尝试读取每一个key-value对，key必须是一个字符串，因此首先进行一个parse_string的操作，parse失败，直接返回
    3. key处理完了，下一个token应该是’:’
    4. 然后对value进行解析。value可以是任意这种类型，因此，直接递归处理，层数+1
    5. 下一个token如果发现是’}’, 说明这个object全部处理完了
    6. 否则应该是’,’, 如果不是，说明有问题
      这部分由于比较核心，还是把代码贴出来，可能会更清晰些。

  5. 对array的处理
    这其实也是一个递归的过程，但是相对比较简单

    1. 首先判断当前这一层是不是一个空的array，已经结束了
    2. 然后尝试解析每一个element，element可以各种类型，因此需要递归处理，同时计算好相应的层数
    3. 遇到’]’, 表示array结束，遇到’,’表示还有下一个element，其他字符，非法情况

然后parse_json函数结束，整个解析过程就结束了。

接下里就是对parse_json做一个简单的封装，作为Json类的成员函数和对外的接口。同时封装了一个parse_multi函数，处理多个json串的解析，虽然我感觉其实意义不大。

最后，就是一个has_shape函数，根据输入的”shape”，判断指定的解析后的json是否有相应的”形状”，但是显然只包含了最外层的判断。比如，可能会输入一个

- shape = [{"name", Json::STRING}, {"age", Json::NUMBER}, {"color", Json::ARRAY}]
- json = {"name": "somename", "age": 20, "color":["blue", "red","yellow"], "language": "English"}

然后发现，shape里各个key对应的类型和实际json里相应key对应的类型是一致的，于是返回true(json里可以多余的其他的key)

这个函数可以对解析后的json 业务上的合法性做一个比较方便的判断（虽然不一定很严格，因为其实没有处理嵌套的情况）

总结

  • 内部各个类之间的依赖非常清晰。
